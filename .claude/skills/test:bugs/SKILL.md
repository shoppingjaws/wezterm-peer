# Your Task

ソースコードを読んで不具合・意図しない挙動を特定し、**正しいあるべき挙動を assert するテスト**として表現する。テストは現在の実装に対して失敗し、修正後に通るようになる。

# Workflow

## Phase 1: コードの理解

1. プロジェクト構造を把握する（CLAUDE.md、src/ 配下のファイル構成）
2. 全ソースコードを読み、ビジネスロジック・データフロー・エッジケースを理解する
3. 既存テストを読み、カバレッジの隙間を把握する

## Phase 2: 不具合の特定

以下の観点でコードを精査する:

- **ロジックエラー**: 条件分岐の抜け、境界値の未処理、off-by-one
- **非対称な挙動**: 類似処理間で一方にだけ存在するチェック・ガード
- **入力バリデーション漏れ**: 不正な入力が通過してしまうケース
- **状態の不整合**: 操作後に返却されるデータが実際の操作結果を反映していない
- **エッジケース**: 空配列、空文字列、null/undefined、型の境界

## Phase 3: テストの作成

1. 既存テストのヘルパー・構造・命名規則に従う
2. 各不具合に対して**正しいあるべき挙動を期待値とするテスト**を書く
   - 現在のバグった挙動ではなく、修正後に通るべき挙動を assert する
3. テストを実行し、**失敗することを確認する**（失敗 = バグの検出に成功）
4. 失敗するテストに `test.skip` を付与してテストスイート全体を green に保つ

## Phase 4: 報告

特定した不具合を以下の形式で報告する:

| # | ファイル:行 | 問題 | 重要度 |
|---|------------|------|--------|
| 1 | file.ts:42 | 説明 | High/Medium/Low |

# Rules

- テストは**正しい挙動**を assert すること。現在のバグった挙動を追認するテストは書かない
- 全ての失敗テストは最終的に `test.skip` にして、テストスイートを green に保つ
- テストファイルは既存のテストディレクトリ構造に配置する
- 既存テストを変更・削除しない
- 設計上の判断（仕様）とバグを区別する。仕様として妥当なものはバグとして報告しない
- 実行時に外部依存（ネットワーク、ファイルシステム等）が不要なユニットテストのみ書く
